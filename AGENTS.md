# Mail Nest 协作与工程约定

## 项目定位

Mail Nest（信匣）是一个面向个人或小团队的邮件收取与 Web 查看系统。系统需要支持用户注册、登录、为每个用户配置多个邮箱账号、自动或手动收取邮件，并在 Web 界面中按用户隔离查看邮件。

## 目录约定

- `mailnest-be/`：后端服务，使用 Golang 开发。
- `mailnest-fe/`：前端应用，使用 Vue 3 + Vite + TypeScript + Ant Design Vue 开发。
- `doc/`：产品需求、架构设计、接口规范、实施计划等中文文档。
- `CHANGELOG.md`：所有需求、设计、代码和文档变更必须记录。
- `README.md`：项目概览、快速开始和关键文档入口。

## 技术栈建议

### 后端

- 语言：Go。
- Web 框架：优先选择轻量、维护活跃的框架，例如 Gin、Echo 或 Chi；实际采用前需在设计文档中说明理由。
- 数据库：SQLite，默认存放在本地数据目录中。
- 本地存储：邮件原文、附件、索引相关文件存放在本地数据目录中。
- 邮件收取：MVP 优先支持 IMAP/IMAPS；POP3、OAuth 邮箱授权、SMTP 发信暂不作为首期必需能力。
- 后台任务：支持手动收取与周期性收取，任务状态和错误需要可查询。

### 前端

- 框架：Vue 3。
- 构建：Vite。
- 语言：TypeScript。
- UI：Ant Design Vue。
- 状态管理：优先 Pinia。
- 路由：Vue Router。
- HTTP：可使用 Axios 或 Fetch 封装，但必须统一处理接口 envelope。

## 多用户与数据隔离

- 系统必须提供用户注册、登录和退出登录能力。
- 每个用户只能访问自己的邮箱配置、收信任务和邮件数据。
- 后端所有涉及用户数据的查询必须带上当前登录用户上下文。
- 邮件账号、邮件、附件、收取任务等核心表必须包含 `user_id` 或可追溯到用户的外键。
- 邮箱密码或授权码必须加密后存入数据库。
- 前端不能依赖隐藏菜单实现隔离，隔离必须由后端鉴权和查询条件保证。

## 统一 JSON 响应格式

所有对外 JSON 接口必须统一使用以下 envelope：

```json
{
  "success": true,
  "message": "",
  "httpCode": 200,
  "data": {}
}
```

- `success` 表示接口处理是否成功。
- `message` 表示给前端显示的说明信息。
- `httpCode` 必须与 HTTP 状态码保持一致。
- `data` 表示业务数据主体，没有数据时返回空对象。
- 新增接口时优先复用统一响应封装，避免每个 handler 手写不同格式。
- 错误响应也必须使用同一 envelope，并设置正确 HTTP 状态码。

## 安全要求

- 密码必须使用安全哈希算法保存，不允许明文保存。
- 邮箱账号密码或授权凭据必须加密后再落盘，禁止明文存入数据库或日志。
- 日志中不得输出用户密码、邮箱密码、授权 token、完整 Cookie 等敏感信息。
- 登录态一期使用 JWT；后续如果要切换 session 或增加 token 黑名单，必须同步更新架构文档。
- 是否开放用户注册由后端 `config.yaml` 控制。
- 所有需要登录的接口必须鉴权。

## 升级与部署安全

- 以后任何升级、部署、迁移、重建容器、调整挂载目录或修改数据库结构，都必须以“不丢用户数据”为最高优先级。
- 生产数据包括但不限于 SQLite 数据库、邮件原文、正文文件、附件文件、邮箱账号配置、OAuth token、规则、文件夹和用户账号。
- 升级前必须确认数据目录、配置文件和数据库文件仍使用持久化挂载，禁止把生产数据放入一次性容器层。
- 涉及数据库 schema、存储路径或数据格式变更时，必须先设计向前兼容迁移方案，并优先补充迁移测试或回归验证。
- 执行生产升级前必须备份真实 `config.yaml`、数据库和本地数据目录；备份文件不得提交到仓库。
- 升级应尽量做到平滑切换：优先使用可重复执行的迁移、健康检查、容器滚动替换或短暂停机替换，避免长时间不可用。
- 升级后必须验证健康检查、登录、邮件列表、邮件详情、附件访问、规则和邮箱账号配置仍可用。
- 如果升级存在不可逆风险，必须先记录回滚方案；没有回滚或备份方案时不得继续生产升级。

## 邮件收取与存储原则

- 系统必须同时保存用户通过邮箱服务器接收和发送的邮件：接收邮件默认来自 `INBOX`，发送邮件来自账号配置的发件箱 IMAP 文件夹。
- 只要邮件已经存在于 IMAP 服务器的收件箱或已发送目录中，手动收取、定时收取或全量同步都应能保存到本地；不能只保存收件箱。
- 不同邮箱服务商的发件箱目录名可能不同，必须支持从 IMAP 服务器目录列表中选择或修正真实发件箱目录，避免固定写死 `Sent`。
- 如果第三方本地客户端发送邮件但没有把已发送邮件保存到服务器发件箱目录，Mail Nest 无法通过 IMAP 拉取该本地-only 邮件；需要在客户端开启“保存到服务器已发送邮件”。
- 邮件收取需要保存必要元数据，例如主题、发件人、收件人、抄送、日期、Message-ID、文件夹、已读状态、附件信息等。
- 邮件正文需要支持纯文本和 HTML 两种内容。
- 附件建议保存到本地文件系统，数据库保存附件元数据和文件路径。
- 原始邮件内容建议保留，方便后续重新解析、排查问题和导出。
- 相同邮箱中的邮件应基于 Message-ID、UID、文件夹等信息做去重。

## 文档要求

- 文档尽可能使用中文。
- 新增重要设计、架构、接口、数据模型或实施计划时，应放到 `doc/` 目录。
- 修改需求、架构或行为时，必须同步更新相关文档。
- 每次变动都必须更新 `CHANGELOG.md`，记录日期、类型和摘要。

## 代码实现约定

- 保持后端、前端边界清晰，前端不直接读取本地数据文件。
- 后端接口路径建议统一以 `/api/v1` 开头。
- 后端应集中封装响应 envelope、错误处理、鉴权中间件和用户上下文。
- 前端应集中封装请求客户端、登录态处理和 envelope 解包逻辑。
- 新增能力时优先补充测试；涉及鉴权、数据隔离、邮件解析、收取去重的逻辑必须重点测试。

## 变更记录

- 每次提交前检查 `CHANGELOG.md` 是否记录本次变动。
- 变更记录建议使用 `新增`、`变更`、`修复`、`文档`、`安全` 等分类。
