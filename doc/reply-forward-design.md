# 邮件回复与转发功能设计

## 1. 目标

回复与转发是在现有 SMTP 发信能力上的增强，目标是让用户在邮件详情中直接完成常见邮件处理动作：

1. 回复发件人。
2. 回复全部相关收件人。
3. 转发邮件给新的联系人。
4. 回复或转发成功后，本地已发送目录可以立即看到发送记录。
5. 收件人、抄送人、转发目标继续自动沉淀到联系人通讯录。

该功能必须继续遵守用户数据隔离、统一 JSON envelope、邮箱凭据加密和升级不丢数据的要求。

## 2. 功能范围

### 2.1 回复

用户在邮件详情点击“回复”后，系统打开写邮件抽屉并自动填充：

- 发件账号：优先使用原邮件所属邮箱账号。
- 收件人：原邮件发件人。
- 主题：如果原主题不是 `Re:` 或 `回复:` 开头，自动追加 `Re: ` 前缀。
- 正文：保留空白编辑区，并在下方追加原邮件引用内容。
- 附件：默认不携带原邮件普通附件。

回复发送时，后端需要在 RFC822 邮件头中写入：

- `In-Reply-To`：原邮件 `Message-ID`。
- `References`：原邮件已有 `References` 加原邮件 `Message-ID`。

如果原邮件没有 `Message-ID`，仍允许发送，但不写入线程头；本地已发送记录仍保存 `source_message_id`，方便 Web 侧追踪来源。

### 2.2 回复全部

用户点击“回复全部”后，系统自动填充：

- 收件人：原邮件发件人。
- 抄送人：原邮件 `To` 和 `Cc` 中除当前用户所有邮箱账号地址以外的联系人。
- 去重：邮箱地址大小写不敏感去重。
- 密送人：不填充。原邮件密送信息通常不可见，也不应被扩散。

回复全部必须避免把当前用户自己的邮箱地址加入收件人或抄送人。判断自己的邮箱地址时，应使用当前用户下所有已配置邮箱账号的 `email` 和 `imap_username` 做大小写不敏感匹配。

### 2.3 转发

用户点击“转发”后，系统打开写邮件抽屉并自动填充：

- 发件账号：优先使用原邮件所属邮箱账号。
- 收件人、抄送、密送：默认为空，由用户填写。
- 主题：如果原主题不是 `Fwd:` 或 `转发:` 开头，自动追加 `Fwd: ` 前缀。
- 正文：插入转发说明头，包含原发件人、收件人、抄送人、日期和主题，再追加原邮件正文。
- 附件：展示原邮件普通附件列表，默认勾选转发；用户可以取消某些附件。

转发普通附件时，后端不能要求前端重新下载再上传原附件，而应支持通过附件 ID 引用当前用户有权限访问的原附件，由后端从本地数据目录读取并重新组包。这样可以减少浏览器流量，也避免大附件在前端重复中转。

## 3. 交互设计

### 3.1 邮件详情动作区

邮件详情标题区域右侧新增一组轻量操作按钮：

- 回复：主按钮或常用按钮。
- 回复全部：当原邮件存在多个参与人时显示。
- 转发：始终显示。

按钮使用图标加中文文字；窄屏或阅读区宽度较小时收进“更多”菜单，避免挤压主题和联系人信息。

### 3.2 写邮件抽屉复用

回复和转发复用现有写邮件抽屉，但需要增加上下文状态：

- `composeMode`：`new`、`reply`、`replyAll`、`forward`。
- `sourceMessageId`：来源邮件 ID。
- `forwardAttachmentIds`：转发时选择的原附件 ID。
- `threadHeaders`：后端预生成或后端发送时根据来源邮件生成。

抽屉标题根据模式显示为：

- 新邮件
- 回复邮件
- 回复全部
- 转发邮件

抽屉底部按钮仍为“取消”和“发送”。发送前校验仍沿用现有发信规则：至少一个收件人、主题或正文或附件不能全部为空、附件数量和总大小不能超过限制。

### 3.3 引用正文

引用正文使用可折叠样式，避免编辑区一打开就被原文淹没：

- 回复：上方是用户输入区域，下方是引用块。
- 转发：上方是用户输入区域，下方是转发头和原文。
- 纯文本邮件使用 `>` 前缀引用。
- HTML 邮件使用 `blockquote` 包裹，并保留基本排版。

安全要求：

- 引用 HTML 进入编辑器前必须经过后端或前端清洗，移除脚本、事件属性和危险协议。
- 邮件详情中用于展示的短期签名内嵌图片 URL 不能直接放入发出的邮件正文，因为外部收件人无法访问，也可能过期。
- 首期可以在引用正文中移除本地内嵌图片并保留文字提示；后续增强可以把原内嵌图片作为 inline 附件重新组包，并改写为新的 `cid:`。

## 4. 后端接口设计

### 4.1 获取写信上下文

`GET /api/v1/messages/{id}/compose-context?mode=reply|replyAll|forward`

该接口根据来源邮件和当前用户账号生成写信初始值，前端不自行拼接复杂收件人和线程头。

响应示例：

```json
{
  "success": true,
  "message": "获取成功",
  "httpCode": 200,
  "data": {
    "mode": "replyAll",
    "sourceMessageId": "123",
    "accountId": "8",
    "to": ["张三 <zhangsan@example.com>"],
    "cc": ["李四 <lisi@example.com>"],
    "bcc": [],
    "subject": "Re: 项目进展",
    "textBody": "\n\n在 2026-07-19 10:20，张三 写道：\n> 原始正文",
    "htmlBody": "<p><br></p><blockquote>原始正文</blockquote>",
    "forwardAttachments": []
  }
}
```

转发模式下 `forwardAttachments` 返回可转发附件：

```json
[
  {
    "id": "45",
    "filename": "report.pdf",
    "contentType": "application/pdf",
    "size": 204800,
    "selected": true
  }
]
```

### 4.2 扩展发信接口

继续复用 `POST /api/v1/messages/send`，在现有 `multipart/form-data` 字段基础上增加可选字段：

- `composeMode`：`new`、`reply`、`replyAll`、`forward`，默认 `new`。
- `sourceMessageId`：来源邮件 ID，回复和转发时必填。
- `forwardAttachmentIds`：JSON 字符串数组，转发原附件时使用。

后端发送前必须校验：

- 来源邮件必须属于当前用户。
- 发件账号必须属于当前用户。
- 转发的附件必须属于来源邮件，且来源邮件属于当前用户。
- 普通上传附件和转发原附件合并后仍受数量和总大小限制。
- 回复和回复全部根据来源邮件生成 `In-Reply-To` 与 `References`，不能信任前端传入线程头。

响应仍使用现有发送成功结构，`data` 返回本地已发送邮件摘要。

## 5. 数据模型设计

为支持线程关系和后续会话聚合，建议给 `mail_messages` 增加兼容字段：

- `in_reply_to`：邮件头 `In-Reply-To`。
- `references_header`：邮件头 `References`。
- `source_message_id`：本地来源邮件 ID，Mail Nest 内发起回复或转发时保存。
- `compose_mode`：`new`、`reply`、`replyAll`、`forward`。

迁移要求：

- 新字段使用 `NULL` 或空字符串默认值，不能重建表或清空邮件。
- IMAP 收取历史邮件时解析并保存 `In-Reply-To` 与 `References`，没有则为空。
- 已有本地邮件无需回填，后续可通过原始邮件修复任务逐步补齐。

## 6. MIME 与附件处理

回复和转发都复用现有 SMTP 发送服务，但需要扩展 `OutgoingMessage`：

- `InReplyTo string`
- `References []string`
- `ForwardedAttachmentIDs []int64` 或在服务层转成 `OutgoingAttachment`
- `ComposeMode string`
- `SourceMessageID int64`

发信原文组包时需要按顺序写入线程头：

1. `From`
2. `To`
3. `Cc`
4. `Subject`
5. `Date`
6. `Message-ID`
7. `In-Reply-To`
8. `References`
9. `MIME-Version`

转发附件读取本地文件时必须做权限校验和路径安全校验，只允许读取数据库中属于当前用户、当前来源邮件的附件路径。

## 7. 联系人与显示规则

回复、回复全部和转发目标都沿用现有联系人逻辑：

- 自动填充时优先显示联系人昵称或姓名，但发送给 SMTP 的地址必须保留真实邮箱。
- 发送成功后，`to`、`cc`、`bcc` 和转发新增目标都自动沉淀联系人。
- 自动沉淀不能覆盖用户手工维护资料。

## 8. 测试要点

后端建议补充：

- 回复生成正确的 `In-Reply-To` 和 `References`。
- 回复全部会排除当前用户自己的邮箱地址并去重。
- 用户不能回复或转发其他用户的邮件。
- 用户不能转发其他邮件或其他用户邮件的附件。
- 转发原附件不需要前端重新上传，发送原文中包含附件。
- 没有 `Message-ID` 的邮件也能回复，只是不写线程头。

前端建议补充或人工验证：

- 邮件详情按钮在宽屏和窄屏不溢出。
- 回复、回复全部、转发打开抽屉后字段填充正确。
- 转发附件可勾选和取消。
- 发送成功后跳转到发件箱并打开新发送邮件。

## 9. 分阶段实现建议

第一阶段：

- 新增写信上下文接口。
- 扩展发信接口支持 `composeMode` 和 `sourceMessageId`。
- 前端详情区增加回复、回复全部、转发入口。
- 回复和转发先不携带原内嵌图片，只保留正文引用和普通附件转发。

第二阶段：

- 解析并保存收取邮件的 `In-Reply-To` 和 `References`。
- 邮件详情展示“这是对某封邮件的回复”或会话关系。
- 转发时支持重新嵌入原内嵌图片并改写 `cid:`。

第三阶段：

- 增加草稿箱和自动保存。
- 增加会话视图，把同一邮件线程聚合展示。
