# Mail Nest 实施计划

## 1. 总体顺序

建议分阶段实现，先完成可运行闭环，再扩展高级能力。

一期已确认决策：

- 登录态使用 JWT。
- 只支持 IMAP/IMAPS。
- 收取 `INBOX` 和可配置的发件箱文件夹。
- 邮箱密码或授权码加密后存入当前配置的数据库。
- 是否开放注册由后端 `config.yaml` 控制。

## 2. 阶段一：项目骨架与基础约定

目标：建立后端、前端基础工程，并落实统一响应格式。

任务：

1. 创建 `mailnest-be/` Go 项目。
2. 创建 `mailnest-fe/` Vue 3 + Vite + TypeScript 项目。
3. 后端实现 `config.yaml` 配置加载、本地数据目录初始化、GORM 数据库连接和 `AutoMigrate` 迁移；默认 SQLite，并兼容 MySQL/PostgreSQL。
4. 后端实现统一 envelope 响应封装。
5. 前端实现 API 客户端和 envelope 解包逻辑。
6. 补充基础 README 和启动说明。

验收：

- 后端健康检查接口可访问。
- 健康检查接口返回统一 envelope。
- 前端可以调用健康检查接口并展示基础页面。

## 3. 阶段二：用户注册登录

目标：实现用户系统和登录态。

任务：

1. 设计并迁移 `users` 表。
2. 实现密码哈希。
3. 实现由 `config.yaml` 控制的注册接口。
4. 实现登录接口。
5. 实现当前用户接口。
6. 实现 JWT 生成和鉴权中间件。
7. 实现前端登录、注册页面。
8. 实现前端路由鉴权。

验收：

- 用户可以注册并登录。
- 未登录用户访问业务页面会被跳转到登录页。
- 登录后可以获取当前用户信息。

## 4. 阶段三：邮箱账号配置

目标：用户可以配置多个邮箱账号。

任务：

1. 设计并迁移 `mail_accounts` 表。
2. 实现邮箱凭据加密和解密。
3. 实现邮箱账号增删改查接口。
4. 实现 IMAP 连接测试接口。
5. 实现前端邮箱账号列表和编辑表单。
6. 实现连接测试和启停状态展示。
7. 增加 SMTP 发信配置字段，密码或授权码按邮箱凭据规则加密保存。

验收：

- 一个用户可以创建多个邮箱账号。
- 邮箱密码或授权码不会明文落库。
- 用户无法访问其他用户的邮箱账号。

## 5. 阶段四：邮件收取

目标：实现 IMAP 邮件拉取和本地存储。

任务：

1. 设计并迁移 `mail_messages`、`mail_attachments`、`mail_sync_jobs` 表。
2. 实现 IMAP 收件箱拉取。
3. 实现邮件 MIME 解析。
4. 实现原始邮件、正文和附件本地落盘。
5. 实现邮件去重。
6. 实现手动触发收取接口。
7. 实现后台周期性收取 worker：每分钟轻量扫描到期启用账号，按账号收取间隔触发日常增量收取，并限制并发避免拖慢服务。
8. 实现收取任务状态记录。
9. 增加 `mail_message_states` 中的垃圾邮件状态，并支持规则动作 `mark_spam`、批量标记/取消标记垃圾邮件和 `systemFolder=spam` 过滤。

验收：

- 用户可以手动触发某个邮箱收取邮件。
- 启用账号到达收取间隔后可以由后端自动触发日常增量收取。
- 收取完成后数据库中出现邮件元数据。
- 本地目录中保存邮件原文、正文和附件。
- 重复收取不会重复插入同一封邮件。

## 6. 阶段五：邮件 Web 查看

目标：用户可以通过 Web 查看邮件。

任务：

1. 实现邮件列表接口。
2. 实现邮件详情接口。
3. 实现附件下载接口。
4. 实现前端邮件列表页面。
5. 实现前端邮件详情页面。
6. 实现按邮箱账号过滤和基础分页。

验收：

- 用户可以查看自己的邮件列表。
- 用户可以打开邮件详情查看正文。
- 用户可以下载附件。
- 用户无法访问其他用户的邮件详情或附件。

## 7. 阶段六：联系人通讯录

目标：用户可以维护联系人，并在邮件列表和详情中优先显示联系人昵称或姓名。

任务：

1. 设计并迁移 `contacts` 表，按 `user_id + email_key` 去重。
2. 实现联系人列表、新增、编辑、删除接口。
3. 邮件收取时从发件人、收件人和抄送人自动沉淀联系人。
4. 自动沉淀只更新最近出现时间和空缺显示名，不覆盖用户手工维护资料。
5. 前端新增联系人页面，支持搜索、分页和维护资料。
6. 邮件列表和详情按邮箱地址匹配通讯录，优先显示昵称或姓名，点击联系人可查看真实邮箱地址和联系信息。

验收：

- 用户可以维护自己的联系人。
- 用户无法访问其他用户联系人。
- 自动发现联系人不会覆盖手工昵称、电话、公司或备注。
- 邮件列表和详情能优先显示联系人名称，详情中可查看完整邮箱地址。

## 8. 阶段七：邮件发送

目标：用户可以通过 Web 界面发送邮件，并在本地已发送目录中立即查看。

任务：

1. 设计并迁移邮箱账号 SMTP 配置字段。
2. 实现 SMTP/SMTPS/STARTTLS 发信服务，支持纯文本和 HTML 正文。
3. 实现 `POST /api/v1/messages/send` 接口，按当前用户校验发件账号归属。
4. 发送成功后保存 RFC822 原文、正文和邮件元数据到本地已发送目录。
5. 发信后自动沉淀收件人、抄送人和密送人为联系人。
6. 前端邮件页新增写邮件抽屉，支持选择发件账号和填写收件人、抄送、密送、主题、正文。

验收：

- 未登录用户不能发信。
- 用户不能使用其他用户的邮箱账号发信。
- SMTP 密码或授权码不会明文落库或回显。
- 发送成功后可以在“发件箱”看到本地保存的邮件。
- 发信相关联系人自动进入通讯录。

## 9. 阶段八：回复与转发

目标：用户可以在邮件详情中直接回复、回复全部和转发邮件。

状态：已完成首期实现，后端已支持写信上下文、线程头和转发附件，前端邮件详情已接入回复、回复全部和转发入口。

任务：

1. 扩展 `mail_messages` 线程字段，保存 `In-Reply-To`、`References`、来源邮件 ID 和回复/转发模式。
2. 扩展 IMAP 解析，保存收取邮件的线程头。
3. 实现 `GET /api/v1/messages/{id}/compose-context`，由后端生成回复、回复全部和转发初始内容。
4. 扩展 `POST /api/v1/messages/send`，支持 `composeMode`、`sourceMessageId` 和 `forwardAttachmentIds`。
5. SMTP 原文组包支持 `In-Reply-To` 和 `References`。
6. 转发原附件时后端按当前用户、来源邮件和附件 ID 校验后读取本地文件并重新组包。
7. 前端邮件详情新增回复、回复全部和转发按钮，窄屏时收进更多菜单。
8. 写邮件抽屉支持回复/转发模式、引用正文和转发附件勾选。

验收：

- 用户不能回复或转发其他用户的邮件。
- 回复邮件的 RFC822 原文包含正确的 `In-Reply-To` 和 `References`。
- 回复全部会排除当前用户自己的邮箱地址，并按大小写不敏感去重。
- 转发原附件不需要前端重复下载上传，且不能跨用户或跨来源邮件引用附件。
- 发送成功后发件箱能看到本地保存的回复或转发邮件。

## 10. 阶段九：邮件批量操作

目标：提升邮件列表整理效率。

任务：

1. 设计并迁移 `mail_message_states` 表。
2. 扩展邮件列表和详情返回本地状态：已读、星标、删除状态。
3. 实现 `POST /api/v1/messages/batch-actions`。
4. 实现 `POST /api/v1/messages/batch-preview`。
5. 前端邮件列表增加多选框和批量工具条。
6. 批量删除加入二次确认，默认只做本地删除。

验收：

- 用户只能批量操作自己的邮件。
- 批量标记、星标、移动和恢复可重复执行且结果稳定。
- 本地删除后默认不展示在普通列表，但可通过回收视图恢复。

## 11. 阶段十：同步日志中心

目标：让用户和维护者能清楚看到同步任务发生了什么。

任务：

1. 扩展 `mail_sync_jobs` 查询接口。
2. 新增 `mail_sync_job_events` 表。
3. 在连接、目录列表、拉取、解析、入库、规则和清理阶段写入事件日志。
4. 实现同步任务详情和事件日志接口。
5. 前端邮箱账号页增加“同步日志”入口。
6. 日志内容做敏感信息脱敏。

验收：

- 同步失败时能在页面看到失败阶段和错误摘要。
- 同步日志不会泄露邮箱密码、OAuth token 或完整 Cookie。
- 单任务日志分页可用，不会一次性加载过多内容。

## 12. 阶段十一：附件中心

目标：支持从附件维度找资料、下载附件和定位原邮件。

任务：

1. 为 `mail_attachments` 增加附件中心查询索引。
2. 实现 `GET /api/v1/attachments`。
3. 附件列表返回所属邮件摘要、账号信息和下载链接。
4. 前端新增附件中心页面。
5. 支持按文件名、类型、账号、文件夹、日期和是否内嵌筛选。
6. 支持从附件中心跳转原邮件。

验收：

- 用户只能看到自己的附件。
- 附件列表分页稳定。
- 附件下载继续复用受鉴权保护的下载接口。

## 13. 阶段十二：规则引擎增强

目标：让规则从基础归档升级为可预览、可排序、可扩展动作的整理工具。

任务：

1. 扩展规则字段：优先级、命中后停止、动作类型。
2. 扩展规则条件：账号、文件夹、附件文件名、附件类型、附件大小、已读和星标。
3. 扩展规则动作：移动文件夹、标记已读、加星。
4. 实现规则预览接口。
5. 手动应用规则支持当前筛选结果。
6. 前端规则编辑器拆分为条件、动作和优先级三段。

验收：

- 规则预览不会修改邮件。
- 多规则命中时按优先级稳定执行。
- 规则执行仍严格按 `user_id` 隔离。

## 14. 阶段十三：完善与加固

目标：提升稳定性、安全性和可维护性。

任务：

1. 增加关键单元测试和集成测试。
2. 完善错误提示和日志。
3. 增加收取任务并发控制。
4. 增加配置示例。
5. 补充部署说明。
6. 检查并更新全部文档。

验收：

- 核心测试通过。
- 常见错误能给出明确提示。
- 文档能指导新开发者启动项目。

## 15. 优先测试点

- 统一 JSON envelope。
- 用户注册、登录、鉴权。
- 用户数据隔离。
- 邮箱凭据加密。
- IMAP 连接测试。
- 邮件去重。
- 附件路径安全。
- 联系人 CRUD 和用户隔离。
- 自动沉淀联系人不覆盖手工维护资料。
- SMTP 发信配置加密、发信后本地保存和联系人沉淀。
- 回复/转发来源邮件和附件的用户隔离。
- 回复全部排除自己的邮箱地址。
- 回复邮件线程头生成正确。
- 邮件批量操作的用户隔离和幂等性。
- 同步日志脱敏和分页。
- 附件中心查询隔离、分页和下载鉴权。
- 规则预览不产生副作用，规则执行按优先级稳定命中。
