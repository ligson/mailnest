# 邮件批量操作、规则增强、同步日志与附件中心设计

## 1. 背景与目标

Mail Nest 目前已经具备邮件收取、查看、发信、联系人、规则归档、附件下载和同步状态记录等基础能力。下一步最实用的增强，不是再扩一批零散按钮，而是把“高频整理邮件”和“高频排查问题”这两件事做扎实。

本设计覆盖四个功能：

- 邮件批量操作。
- 规则引擎增强。
- 同步日志中心。
- 附件中心。

目标是让用户能更快整理邮件、更容易定位同步问题，并能从附件维度回看资料。

## 2. 设计原则

- 以当前登录用户为唯一边界，所有查询和写入都必须带 `user_id`。
- 复用现有邮件、附件、规则和同步任务基础表，优先做增量扩展。
- 邮件内容尽量保持不可变，用户操作类状态单独管理。
- 所有对外接口继续使用统一 JSON envelope。
- 日志只记录排障所需信息，不能输出密码、token、完整 Cookie 或邮箱敏感内容。

## 3. 邮件批量操作

### 3.1 目标场景

- 批量标记已读、未读。
- 批量加星、取消加星。
- 批量移动到本地文件夹。
- 批量删除到本地回收区，保留邮件数据。
- 从筛选结果中一次处理多封邮件。

### 3.2 数据模型

建议新增 `mail_message_states`，把“邮件内容”与“用户操作状态”拆开：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `user_id` | 用户 ID |
| `message_id` | 邮件 ID |
| `is_read` | 是否已读 |
| `read_at` | 最近一次标记已读时间 |
| `starred` | 是否加星 |
| `is_spam` | 是否被标记为垃圾邮件 |
| `spam_at` | 最近一次标记为垃圾邮件的时间 |
| `archived_at` | 归档时间 |
| `deleted_at` | 本地删除时间 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

约束：

- `user_id + message_id` 唯一。
- 批量操作只改状态，不改邮件原文、正文和附件文件。
- 当前版本不回写远端 IMAP 状态，默认按本地视图生效。

### 3.3 批量操作类型

建议支持：

- `mark_read`
- `mark_unread`
- `star`
- `unstar`
- `mark_spam`
- `unmark_spam`
- `move_folder`
- `delete`
- `restore`

其中 `delete` 默认表示本地删除，不直接删除服务器邮件；后续如需要“同步删除远端邮件”，应作为显式高级开关单独设计。

### 3.4 API 设计

```text
POST /api/v1/messages/batch-actions
GET  /api/v1/messages/batch-preview
```

批量操作请求示例：

```json
{
  "messageIds": ["1", "2", "3"],
  "action": "move_folder",
  "folderId": "10"
}
```

批量预览用于在执行前估算影响范围：

```json
{
  "messageIds": ["1", "2", "3"]
}
```

### 3.5 前端交互

- 邮件列表左侧增加多选框。
- 列表顶部增加批量工具条。
- 选中多封邮件后显示动作计数。
- 批量移动时弹出文件夹选择器。
- 删除前弹出二次确认。

## 4. 规则引擎增强

### 4.1 当前定位

现有规则已经能按条件把邮件放入本地文件夹。下一步重点不是堆复杂布尔表达式，而是把“更好配、可预览、可复用、可扩展动作”补齐。

### 4.2 条件增强

建议在现有字段基础上补充：

- 发件账号 `account_id`。
- 所属系统文件夹 `folder`。
- 附件文件名。
- 附件类型。
- 附件大小范围。
- 是否已读。
- 是否已星标。

这样可以覆盖更常见的“工作流归档”和“通知聚类”场景。

### 4.3 动作增强

建议规则动作从单一“移动文件夹”扩展为：

- 移动到本地文件夹。
- 标记已读。
- 加星。
- 标记为垃圾邮件，从默认邮件列表和收件箱视图隐藏。
- 归档到本地文件夹后从收件箱视图隐藏。

仍建议保持“单规则单主动作”的简洁风格，避免一下子变成复杂流程编排器。

### 4.4 规则执行策略

- 新邮件入库后自动执行。
- 手动“重新应用规则”时支持按范围执行：
  - 只处理未归档邮件。
  - 处理当前筛选结果。
  - 处理全部历史邮件。
- 多条规则命中时按优先级从小到大执行。
- 支持“命中后停止继续匹配”。

### 4.5 API 设计

```text
GET  /api/v1/mail-rules
POST /api/v1/mail-rules
PUT  /api/v1/mail-rules/{id}
DELETE /api/v1/mail-rules/{id}
POST /api/v1/mail-rules/apply
POST /api/v1/mail-rules/preview
```

建议新增字段：

- `priority`：规则优先级。
- `stopOnMatch`：命中后停止。
- `actionType`：`move_folder` / `mark_read` / `star` / `mark_spam`。

### 4.6 前端交互

- 规则列表展示命中条件摘要和动作摘要。
- 规则编辑器用“条件区 + 动作区 + 优先级区”三段式布局。
- 提供“测试匹配”功能，输入一封样例邮件后展示命中结果。

## 5. 同步日志中心

### 5.1 目标场景

- 用户想知道某个邮箱为什么没收下来。
- 用户想看同步卡在哪一步。
- 用户想知道全量同步做了多少、失败在哪封邮件。
- 用户想重试或继续排查。

### 5.2 数据模型

现有 `mail_sync_jobs` 保留任务汇总信息，建议新增 `mail_sync_job_events` 记录过程日志：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `job_id` | 同步任务 ID |
| `level` | `info` / `warn` / `error` |
| `phase` | `connect` / `list` / `fetch` / `parse` / `store` / `rule` / `cleanup` |
| `message` | 简短日志 |
| `detail_json` | 结构化详情，便于前端展示 |
| `created_at` | 创建时间 |

建议：

- 单个任务日志默认保留最近 200 条，避免无限增长。
- 结构化详情里不要写敏感账号信息。

### 5.3 API 设计

```text
GET /api/v1/mail-accounts/{id}/sync-jobs
GET /api/v1/mail-sync-jobs/{jobId}
GET /api/v1/mail-sync-jobs/{jobId}/events
```

返回内容建议包含：

- 任务类型：手动、定时、全量。
- 任务状态：运行中、成功、失败、取消。
- 当前阶段和最近一条日志。
- 新增邮件数、处理数、错误数。

### 5.4 前端交互

- 邮箱账号卡片显示最近一次同步状态。
- 点击“同步日志”打开抽屉或弹窗。
- 日志按时间倒序展示，错误高亮。
- 全量同步详情里显示进度和最近阶段。

## 6. 附件中心

### 6.1 目标场景

- 按文件名快速找附件。
- 按类型筛出 PDF、图片、压缩包、文档。
- 查看某段时间内收到的附件。
- 从附件反查原邮件。

### 6.2 数据模型

现有 `mail_attachments` 已保存附件元数据，可直接复用。建议补充查询索引：

- `user_id + filename`
- `user_id + content_type`
- `user_id + created_at`
- `user_id + inline`

如果后续附件量特别大，再考虑独立附件搜索索引表。

### 6.3 API 设计

```text
GET /api/v1/attachments
GET /api/v1/attachments/{id}
```

列表查询支持：

- `keyword`：文件名模糊搜索。
- `contentType`：按 MIME 类型过滤。
- `accountId`：按邮箱账号过滤。
- `folderId`：按本地文件夹过滤。
- `inline`：是否内嵌附件。
- `dateFrom` / `dateTo`：按邮件时间过滤。
- `page` / `pageSize`。

列表项建议返回：

- 附件名、类型、大小。
- 所属邮件主题、发件人、时间。
- 所属账号和文件夹。
- 下载链接和原邮件链接。

### 6.4 前端交互

- 新增“附件中心”页面。
- 顶部提供搜索和筛选条。
- 列表项支持直接下载、定位邮件、复制文件名。
- 图片和 PDF 可做轻量预览，其他类型只展示元数据。

## 7. 建议实施顺序

1. 先做邮件批量操作，因为它最直接提升列表页效率。
2. 再做同步日志中心，因为它最能减少排障成本。
3. 然后做附件中心，因为它复用现有附件数据最充分。
4. 最后做规则引擎增强，因为它会影响条件编辑器和执行逻辑，联动面最大。

## 8. 风险控制

- 批量删除必须默认本地删除，不回写远端。
- 同步日志必须做敏感信息脱敏。
- 附件中心分页要稳，不能一次性把所有附件拉到前端。
- 规则增强要保留简单模式，避免把普通用户逼进复杂流程编排。
